# 자동 변환 규칙 (Auto-Transform Rules)

> 이 파일은 Cline이 사용자 요청을 패턴 매칭하여 구조화하는 기준입니다.
> 개발자가 직접 입력한 프롬프트를 아래 규칙에 따라 내부적으로 확장하여 처리합니다.

## 우선순위

1. 사용자가 `[의도]`, `[범위]` 등을 직접 명시하면 이 파일을 **무시**하고 사용자 구조 우선
2. 아래 패턴 매칭되면 해당 템플릿 적용
3. 모두 해당 없음 → 기본 응답 규칙(한국어, 3줄 요약) 적용

---

## 패턴 1: 리팩토링 (이동/변경/최적화)

- **트리거**: `리팩토링`, `이동`, `옮겨`, `변경`, `패키지`, `최적화`, `성능`
- **내부 변환**:
  - 의도: {대상} → {목적지} 경로/패키지 변경
  - 범위: 해당 모듈 전체 (Controller, Service, Mapper, DTO, XML, React)
  - 출력: 변경된 경로/패키지명만 3줄 요약
  - 단계: 영향분석 → 생성 → 삭제 → 참조수정 → 빌드

## 패턴 2: 버그 수정 (에러/디버깅)

- **트리거**: `버그`, `에러`, `오류`, `고쳐`, `안돼`, `문제`, `터져`
- **내부 변환**:
  - 의도: 오류 원인 파악 및 수정
  - 범위: 해당 파일 + 연관 import/호출부
  - 출력: 원인 분석 3줄 + 수정 코드 (diff 형식)
  - 단계: 원인추적 → 수정 → 검증(컴파일)

## 패턴 3: 신규 기능 개발

- **트리거**: `생성`, `추가`, `만들어`, `개발`, `기능`, `보일러플레이트`
- **내부 변환**:
  - 의도: 신규 CRUD/모듈 생성
  - 범위: 언급된 도메인 경로
  - 출력: 생성된 파일 목록 + 주요 코드
  - 단계: 설계(구조선택) → 파일생성 → 구현 → 테스트

## 패턴 4: 코드 분석/설명

- **트리거**: `설명`, `찾아`, `어떻게`, `위치`, `리뷰`
- **내부 변환**:
  - 의도: 해당 파일/기능 구조 파악
  - 범위: 언급된 파일 또는 현재 탭
  - 출력: 3줄 이내 핵심 요약 (코드 생략 가능)
  - 단계: 탐색 → 분석 → 요약 (수정 금지)

## 패턴 5: 세션 복구

- **트리거**: `끊겼`, `이어서`, `다시`, `복구`
- **내부 변환**:
  - 의도: 직전 작업 중단 지점 파악 및 재개
  - 범위: 현재 Task 히스토리 전체
  - 출력: 마지막 단계 + 남은 단계 요약
  - 단계: 미완료 지점부터 즉시 재개

## 패턴 6: 테스트 (단위/Mock)

- **트리거**: `테스트`, `Mock`, `단위테스트`, `더미`
- **내부 변환**:
  - 의도: 테스트 코드 또는 Mock 데이터 생성
  - 범위: 언급된 함수/API
  - 출력: 테스트 코드 블록 (Jest/Pytest 등)
  - 단계: 분석 → 테스트코드 생성 → 엣지케이스 포함

## 패턴 7: 문서화 (주석/README/다이어그램)

- **트리거**: `문서화`, `주석`, `README`, `다이어그램`, `시각화`
- **내부 변환**:
  - 의도: 코드 문서화 또는 시각화
  - 범위: 해당 파일 또는 프로젝트 전체
  - 출력: 생성된 문서/다이어그램 (Mermaid 등)
  - 단계: 분석 → 문서생성 → 정리
