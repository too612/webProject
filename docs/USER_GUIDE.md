# GitHub Copilot & DeepSeek 병행 사용 가이드 (USER_GUIDE.md)

본 문서는 GitHub Copilot과 DeepSeek(Cline)을 상황에 맞게 병행 사용하는 개발자를 위한 최적화 가이드입니다.  
두 도구는 **컨텍스트 수집 방식**과 **토큰 비용 구조**가 완전히 다르므로, 각 도구에 맞는 전략을 숙지해야 합니다.

## 🚨 도구별 핵심 차이 (반드시 숙지)

| 항목                 | GitHub Copilot                            | DeepSeek + Cline                                         |
| :------------------- | :---------------------------------------- | :------------------------------------------------------- |
| **컨텍스트 수집**    | 에디터에 **열린 모든 탭**을 자동으로 흡수 | **`read_file`로 명시적으로 읽은 파일만** 컨텍스트로 사용 |
| **토큰 과소비 주범** | 무관한 탭을 많이 열어둠                   | 불필요한 `read_file` 호출 또는 방대한 파일을 매번 읽음   |
| **시스템 지시문**    | `copilot-instructions.md` 참조 (자동)     | `.clinerules` → `copilot-instructions.md` 참조 (자동)    |
| **컨텍스트 창**      | 모델에 따라 다름 (일반적으로 200K)        | **1M 토큰** (매우 넉넉)                                  |
| **비용 구조**        | 구독료 (무제한)                           | **종량제** (토큰 사용량 × 단가)                          |
| **캐시 활용**        | 없음 (매 요청 새로 계산)                  | **Prefix 캐싱** (동일 접두사 시 98% 할인)                |
| **응답 속도**        | 빠름                                      | 상대적으로 느림 (API 호출)                               |

## 1. GitHub Copilot 사용 시 최적화 전략

### 1-1. 무관한 에디터 탭 닫기 (가장 중요)

> ⚠️ Copilot은 현재 활성화된 파일뿐만 아니라, **에디터에 열려 있는 모든 탭의 코드를 컨텍스트로 자동 흡수**합니다.

- 관련 없는 파일이 많이 열려 있으면 질문 한 줄에도 수만 토큰이 낭비됩니다.
- **단축키**: `Ctrl + K W` (전체 탭 닫기) → 필요한 파일만 다시 열기

### 1-2. `copilot-instructions.md` 슬림하게 유지

- Copilot은 이 파일을 **매 질문마다** 참조하므로, 20줄 미만으로 유지해야 기본 소모 토큰을 줄일 수 있습니다.
- 상세 규칙은 `SYSTEM-ARCHITECTURE.md`, `FRONT-FULLSTACK-SPEC.md`로 분리하고, 필요 시 `#파일명`으로 태깅하세요.

### 1-3. Copilot 전용 프롬프트 템플릿

[작업]

- 신규 메뉴 'erp/account/slip' (회계 전표 관리) 기능 개발 시작

[참조 파일]

- #SYSTEM-ARCHITECTURE.md
- #FRONT-FULLSTACK-SPEC.md

[요청 사항]

1. 두 문서의 규칙에 따라 생성해야 할 Front 4개 파일, Back 4개 파일, XML 1개 파일의 정확한 경로와 파일명을 리스트업해줘.
2. FRONT-FULLSTACK-SPEC.md의 [3. API 계약 규칙]을 참고하여, 수정(setUpdate/{id})과 소프트 삭제(delRemove/{id})가 규격대로 반영된 코드만 생성해줘. (설명 생략)

## 2. DeepSeek + Cline 사용 시 최적화 전략

### 2-1. 탭 관리 무의미 → `read_file` 제어가 핵심

> ✅ Cline은 **에디터에 열린 탭과 무관**하게, 오직 `read_file`로 읽은 파일만 컨텍스트에 포함됩니다.

- 불필요한 파일을 읽지 않도록 **`copilot-instructions.md`에 "관련 없는 전체 탐색 금지" 규칙**을 이미 추가했으므로 OK.
- 필요할 때만 `#파일명`으로 명시적 태깅하여 읽히도록 유도하세요.

### 2-2. 캐시 히트를 고려한 프롬프트 설계 (비용 절감 핵심)

> 💰 DeepSeek은 **Prefix Matching(접두사 완전 일치)** 으로 캐시를 판단합니다.  
> 동일한 접두사가 반복되면 입력 토큰 비용이 **98% 할인**됩니다.

**캐시 히트를 위한 3가지 원칙**:

1. **고정 접두사**를 매 요청마다 동일하게 유지하세요.
2. **가변적인 질문**은 접두사 뒤에 붙이세요.
3. **참조 파일 순서**를 항상 동일하게 유지하세요.

**DeepSeek 전용 프롬프트 템플릿**:

[고정 접두사 - 매 요청 동일하게 유지 (캐시 히트 유도)]

- 프로젝트 규칙: #SYSTEM-ARCHITECTURE.md, #FRONT-FULLSTACK-SPEC.md 준수
- 응답 형식: 요약(3줄 이내) + 변경된 코드 블록만
- 불필요한 @workspace 탐색 금지

[가변 영역 - 실제 작업 내용]
[작업] 신규 메뉴 'erp/account/slip' 개발
[요청] 두 문서의 규칙에 따라 Front/Back 뼈대 코드 생성

### 2-3. `infinite` 모드 사용 시 주의

- DeepSeek은 컨텍스트 창이 1M로 넉넉하지만, **너무 많은 히스토리를 쌓으면 응답 지연**이 발생할 수 있습니다.
- 장기 대화 시에는 **새 세션을 시작**하거나, `#요약` 명령으로 컨텍스트를 압축하세요.

### [템플릿 C] 일상 작업용 간편 프롬프트 (Daily Prompt)

> 템플릿 A와 B는 구조적이고 복잡한 작업을 위한 것입니다.  
> **버그 수정, 코드 리뷰, 단순 기능 추가, 파일 위치 확인** 등 일상적인 작업에서는 아래 **3단축 공식**만 기억하세요.

#### 🎯 3단축 공식

#### 📋 실제 사용 예시 (복붙 가능)

| 상황              | 이렇게만 써도 OK                                                            |
| :---------------- | :-------------------------------------------------------------------------- |
| **새 기능 개발**  | `erp/account/slip 목록 조회 API랑 화면 만들어줘. 4파일 세트로.`             |
| **버그 수정**     | `SlipController.java에서 insert가 안되는데, 파라미터 바인딩 문제인지 봐줘.` |
| **코드 설명**     | `지금 이 pastorHook.ts 로직이 어떻게 흘러가는지 설명만 해줘.`               |
| **작은 리팩토링** | `sermonApi.ts에 있는 중복 에러 처리, common/api로 함수 빼줘.`               |
| **단순 질문**     | `이 프로젝트에서 FileUploadUtil은 어디에 있어?`                             |
| **코드 리뷰**     | `방금 내가 작성한 SlipService.java 한번만 봐주고 문제 있으면 알려줘.`       |

#### ⚠️ 도구별 미세 팁

| 구분        | GitHub Copilot                       | DeepSeek + Cline                       |
| :---------- | :----------------------------------- | :------------------------------------- |
| **파일명**  | `SlipController.java` (그냥 이름만)  | `#SlipController.java` (태깅 추천)     |
| **경로**    | `erp/account/` 폴더라고만            | `erp/account/ 하위만` (범위 좁히기)    |
| **탭 관리** | **질문 전에 관련 없는 탭 모두 닫기** | 신경 쓸 필요 없음 (`read_file`만 읽음) |

> 💡 **꿀팁**: 대화 초반에 *"이번 대화에서는 파일 수정 시 항상 영향받는 파일 목록을 먼저 보여줘"*라고 한 번만 말하면, 이후로는 위 예시처럼 짧게 말해도 AI가 그 규칙을 기억하고 적용합니다.

## 3. 도구별 전략 비교 (한눈에 보기)

| 상황                  | GitHub Copilot                                                      | DeepSeek + Cline                                             |
| :-------------------- | :------------------------------------------------------------------ | :----------------------------------------------------------- |

## 4. 신규 대분류 도메인 추가 시 (공통 절차)

> 두 도구 모두 동일하게 적용되는 **프로젝트 구조 변경** 절차입니다.

### 4-1. 기준 문서(MD)에 새 도메인 명시

- `SYSTEM-ARCHITECTURE.md` → [2. 도메인 구성] 테이블에 새 도메인 정보와 `[진행 예정]` 태그 추가
- `FRONT-FULLSTACK-SPEC.md` → [1. 목적 및 범위]의 대상 도메인 목록에 새 도메인명 추가

### 4-2. DB 메뉴 시드(`erp_menu_seed.sql`) 경로 등록

- 새 도메인 화면의 접근 라우팅 경로를 SQL 시드 파일에 먼저 정의합니다.

### 4-3. 물리 폴더 생성 (역할 기반 하위 폴더 금지)

- 백엔드: `src/main/java/com/main/app/신규도메인/기능/`
- 프런트엔드: `frontend/src/신규도메인/기능/`
- **`controller/`, `service/` 같은 하위 폴더는 절대 생성하지 마세요.**

### 4-4. 첫 번째 기능 세트 보일러플레이트 요청

- 위의 **도구별 프롬프트 템플릿**을 사용하여 뼈대 코드를 생성합니다.

## 7. 자주 묻는 질문 (FAQ)

### Q1. Copilot과 DeepSeek을 같은 프로젝트에서 번갈아 써도 되나요?

**A.** 네, 가능합니다. 다만 **`copilot-instructions.md`는 두 도구 모두 동일하게 적용**되므로, AI의 행동 규칙은 일관되게 유지됩니다. 단, 프롬프트 템플릿은 도구별로 다르게 사용하세요.

### Q2. DeepSeek에서 캐시 히트율을 확인할 수 있나요?

**A.** DeepSeek API 응답 헤더에 `X-Cache: Hit` 또는 `X-Cache: Miss`가 포함되어 있습니다. Cline이나 커스텀 클라이언트에서 이 값을 로깅하면 캐시 효율을 모니터링할 수 있습니다.

### Q3. Copilot 사용 중에도 `USER_GUIDE.md`를 참조해야 하나요?

**A.** 네. `USER_GUIDE.md`는 **개발자 행동 가이드**이므로, 어떤 도구를 쓰든 프롬프트 작성법과 프로젝트 규칙 숙지에 도움이 됩니다. 다만, **도구별 최적화 섹션(1, 2장)**은 자신이 현재 사용 중인 도구에 맞게 참고하세요.

> 💡 **AI 자동 변환 규칙은 별도 파일(`.ai/AI_AUTO_RULES.md`)로 분리되었습니다.**  
> 개발자는 이 가이드의 **"일상 작업용 간편 프롬프트"** 섹션(2장)에 따라 자연스럽게 질문하면 됩니다.
