# GitHub Copilot 토큰 최적화 및 프로젝트 활용 가이드 (USER_GUIDE.md)

본 문서는 GitHub Copilot 요금제 개편에 따라 효율적인 토큰(Token) 관리와 개발 생산성 유지를 위해 작성되었습니다. 우리 프로젝트의 핵심 기준 문서인 SYSTEM-ARCHITECTURE.md 및 FRONT-FULLSTACK-SPEC.md와 유기적으로 연동하여 최상의 결과물을 도출하는 지침을 제공합니다.

---

## 1. Copilot 토큰 다이어트 핵심 요약

### 1-1. 무관한 에디터 탭 닫기 (가장 중요)

- Copilot은 현재 활성화된 파일뿐만 아니라, 에디터에 열려 있는 모든 탭의 코드를 컨텍스트로 자동 흡수합니다.
- 관련 없는 파일이 많이 열려 있으면 질문 한 줄에도 수만 토큰이 허공에 날아갑니다. 다른 기능을 작업할 때는 반드시 Ctrl + K W를 눌러 전체 탭을 닫은 후 필요한 파일만 열어주세요.

### 1-2. 개발자 가이드와 지시문 파일의 역할 분리

- copilot-instructions.md는 AI가 매 질문마다 읽는 '시스템 지시문'이므로 20줄 미만으로 극도로 슬림하게 유지해야 기본 소모 토큰을 아낄 수 있습니다.
- 프로젝트 전체 구조나 상세 비즈니스 제약사항, 템플릿 등은 이 가이드라인 문서(USER_GUIDE.md) 하단이나 별도 파일에 격리해 두고 필요할 때만 AI에게 참조(태깅)시켜야 합니다.

## 2. 프로젝트 기준 문서 연계 프롬프트 작성법

프로젝트 전체 스캔(@workspace)을 돌리면 엄청난 토큰이 소모되며 아키텍처 규칙이 깨진 코드가 나올 확률이 높습니다. 대신, 아래 템플릿을 활용하여 기준 문서를 명시적으로 태깅(#)하여 질문하세요.

### [템플릿 A] 신규 기능/메뉴 개발용 보일러플레이트 서식

새로운 메뉴나 기능을 시작할 때 아래 서식을 복사해서 사용하면 SYSTEM-ARCHITECTURE.md와 FRONT-FULLSTACK-SPEC.md에 명시된 규칙(4파일 세트 구조, URL 패턴, 소프트 삭제)을 완벽히 준수하는 뼈대 코드를 단번에 얻을 수 있습니다.

(복사 시작)
[작업]

- 신규 메뉴 'erp/account/slip' (회계 전표 관리) 기능 개발 시작

[참조 파일]

- #SYSTEM-ARCHITECTURE.md
- #FRONT-FULLSTACK-SPEC.md

[요청 사항]

1. 두 문서의 규칙에 따라 생성해야 할 Front 4개 파일, Back 4개 파일, XML 1개 파일의 정확한 경로와 파일명을 리스트업해줘.
2. FRONT-FULLSTACK-SPEC.md의 [3. API 계약 규칙]을 참고하여, 수정(setUpdate/{id})과 소프트 삭제(delRemove/{id})가 규격대로 반영된 SlipController.java와 slipApi.ts 뼈대 코드만 최소한으로 생성해줘. (설명 생략)
   (복사 끝)

### [템플릿 B] 기존 기능 버그 수정 및 개선 서식

특정 도메인의 에러를 잡거나 기능을 고칠 때 AI가 엉뚱한 파일이나 폴더를 건드리지 못하도록 제약 조건을 거는 템플릿입니다.

(복사 시작)
[작업]

- 결제 완료 후 마이페이지 진입 시 NullPointerException 발생하는 에러 수정

[참조 파일]

- #MypageRoutes.tsx
- #myuserHook.ts
- #FRONT-FULLSTACK-SPEC.md (엔드포인트 및 응답 규격 참고용)

[제약 조건]

- 위 지정된 파일 외에 프로젝트 전체를 탐색하지 말 것.
- 변경된 코드 블록(수정된 라인 위주)만 출력하고 장황한 설명은 생략할 것.
- 기존의 코드 스타일 및 주석 규칙을 엄격히 유지할 것.

## 3. [확장 규칙] 신규 대분류 도메인을 추가해야 할 때

새로운 대분류 도메인(예: shop, academy 등)을 프로젝트에 아예 새로 추가해야 하는 상황이 오면, 코드부터 작성하지 말고 아래 4단계 절차에 따라 기준 문서를 먼저 선언한 뒤 개발을 진행해야 토큰 낭비와 오작동을 막을 수 있습니다.

### 3-1. 기준 문서(MD)에 새 도메인 명시

Copilot이 새로운 도메인의 존재를 인지할 수 있도록 프로젝트 내 핵심 문서 2곳에 텍스트를 먼저 추가합니다.

- SYSTEM-ARCHITECTURE.md -> [2. 도메인 구성] 테이블에 새 도메인 정보와 [진행 예정] 태그를 추가합니다.
- FRONT-FULLSTACK-SPEC.md -> [1. 목적 및 범위]의 대상 도메인 목록에 새 도메인명을 추가합니다.

### 3-2. DB 메뉴 시드(erp_menu_seed.sql) 경로 등록

- SYSTEM-ARCHITECTURE.md 대원칙에 따라, 새 도메인 화면의 접근 라우팅 경로를 SQL 시드 파일에 먼저 정의하거나 insert 합니다. 이를 통해 AI가 잘못된 프런트 URI를 생성하는 것을 차단합니다.

### 3-3. 물리 폴더 생성 (하위 폴더 금지 규칙 준수)

- 백엔드와 프런트엔드에 동일한 이름으로 새 도메인 폴더를 생성합니다.
- 이때 copilot-instructions.md 규칙에 따라 역할 기반 하위 폴더(controller/, service/)는 절대로 생성하지 않고, 파일들을 패키지/폴더 내에 수평적으로 위치시킵니다.
  - 백엔드 경로: src/main/java/com/main/app/신규도메인/기능/
  - 프런트엔드 경로: frontend/src/신규도메인/기능/

### 3-4. 첫 번째 기능 세트 보일러플레이트 요청

- 폴더 구조가 잡히면 위의 [신규 기능/메뉴 개발용 템플릿]을 사용하여 Copilot에게 해당 도메인의 첫 뼈대 코드를 요청합니다. 반드시 새 도메인 폴더에 수평적으로 패키지가 정의되도록 지시합니다.
